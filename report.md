# Technical Report: Semantic Search Engine with PostgreSQL and pgvector

## Introduction

This project implements a semantic document retrieval system that returns documents by meaning instead of exact term overlap. The system stores text documents and dense vector embeddings in PostgreSQL with the pgvector extension. At query time, a natural-language query is embedded with the same Sentence-Transformers model used during ingestion, then compared against stored vectors with cosine distance. A TF-IDF retrieval engine is included as a classical baseline for comparison.

The main problem is vocabulary mismatch. A keyword engine can fail when a document and a query use different words for the same concept. Semantic retrieval reduces this problem by mapping text into a vector space where related meanings are close even when their surface forms differ.

The selected dataset is AG News from Hugging Face. It contains approximately 120,000 English news documents across World, Sports, Business, and Sci/Tech categories. This exceeds the minimum requirement of 1,000 text documents and provides short, topical documents suitable for both semantic and lexical retrieval experiments.

## State of the Art: Vector Databases

Vector databases store high-dimensional numeric representations of data and provide similarity search over those vectors. In text retrieval, vectors are usually embeddings generated by a neural language model. Similarity is commonly measured with cosine similarity, dot product, or Euclidean distance.

PostgreSQL with pgvector is used in this project because it combines relational storage with vector search. This allows metadata, source fields, document content, and embeddings to remain in one database. pgvector supports vector data types, vector operators, and indexes such as IVFFlat for approximate nearest-neighbor search.

A vector database is useful when retrieval depends on meaning, not exact tokens. Typical use cases include semantic search, recommendation, duplicate detection, clustering, retrieval-augmented generation, and multilingual search. It is less suitable when exact filtering, Boolean matching, or deterministic term matching is the primary requirement. In those cases, lexical indexes may be preferable or should be combined with vector search.

## Methodology

### Dataset Selection

AG News was selected for the following reasons:

- It contains more than 1,000 documents.
- It is public and available through Hugging Face datasets.
- It contains diverse topics that support qualitative search evaluation.
- It is small enough per document to embed efficiently with a compact transformer model.
- It supports category-level analysis through labels stored in JSON metadata.

Each row is transformed into a document with a title, content, source, and metadata field. The title is derived from the first sentence prefix. The content is cleaned before embedding and database insertion.

### Text Preprocessing

The preprocessing pipeline normalizes Unicode text, strips HTML tags, collapses whitespace, and trims leading or trailing spaces. It also provides helper functions for chunking long text and truncating output snippets. Chunking is included to support longer datasets even though AG News documents are short.

### Vectorization Method

The embedding model is `all-MiniLM-L6-v2` from Sentence-Transformers. It produces 384-dimensional embeddings and is optimized for semantic textual similarity. The model is compact, open-source, and fast enough for local experimentation. Embeddings are normalized so cosine similarity can be interpreted consistently.

## Implementation Details

### Repository Components

The source code is organized into clear modules:

- `src/utils/preprocess.py` contains text cleaning, chunking, and truncation utilities.
- `src/embeddings/encoder.py` wraps Sentence-Transformers model loading and embedding generation.
- `src/db/connection.py` manages SQLAlchemy and psycopg2 database connections.
- `src/db/models.py` defines ORM models and bulk insert helpers.
- `src/search/semantic.py` performs pgvector cosine search.
- `src/search/classical.py` implements the TF-IDF baseline.
- `scripts/ingest.py` downloads, preprocesses, embeds, and stores documents.
- `scripts/search.py` exposes an interactive and single-query CLI.
- `scripts/evaluate.py` runs validation queries and writes evaluation artifacts.

### Database Schema

The schema has two main tables. The `documents` table stores raw text and metadata. The `embeddings` table stores one vector per document and model name. The separation supports re-ingesting embeddings from multiple models without duplicating document content.

The schema enables pgvector with `CREATE EXTENSION IF NOT EXISTS vector`. The embedding column uses `VECTOR(384)`, matching the selected Sentence-Transformers model. An IVFFlat index with cosine operators accelerates nearest-neighbor retrieval.

### Ingestion Pipeline

The ingestion script performs these steps:

1. Load AG News from Hugging Face.
2. Clean each document.
3. Insert documents into PostgreSQL.
4. Generate embeddings in batches.
5. Insert embeddings in chunks to avoid oversized transactions.

The default ingestion limit is 5,000 documents, which exceeds the minimum requirement while keeping the initial run practical. The limit can be increased to the full dataset or any supported subset.

### Query Pipeline

The semantic search pipeline performs these steps:

1. Accept a query string from the CLI.
2. Encode the query with the configured embedding model.
3. Convert the vector to pgvector input format.
4. Execute an SQL query ordered by cosine distance.
5. Return ranked results with cosine similarity scores.

The TF-IDF pipeline loads documents into memory, builds a scikit-learn vectorizer with unigrams and bigrams, and ranks documents by cosine similarity between sparse TF-IDF vectors.

## Results

The evaluation script runs ten representative queries across business, science, politics, sports, environment, medicine, and security topics. For each query, it records semantic search latency, TF-IDF search latency, overlap between the top-k result sets, and average score per method.

Outputs are written to:

- `data/evaluation_results.csv`
- `data/evaluation_results.png`

The expected qualitative result is that semantic search can retrieve conceptually related documents even when query terms do not exactly match the document terms. TF-IDF usually performs well when important query tokens appear in the document. Result overlap can be low because the two methods optimize different relevance signals.

## Performance Analysis

Semantic search performance has two major costs: query embedding generation and vector retrieval. Query embedding depends on the Sentence-Transformers model and hardware. pgvector retrieval depends on table size, index configuration, PostgreSQL memory settings, and whether approximate indexing is used effectively.

TF-IDF performance has a different profile. Building the TF-IDF index requires loading all documents into memory, but individual queries are fast for moderate dataset sizes. This approach is simpler but less scalable when documents are large or frequently updated.

The IVFFlat index improves vector search scalability for larger datasets. For production use, the index should be created after a representative number of embeddings have been inserted, and PostgreSQL settings should be tuned for the expected workload.

## Critical Discussion

The system satisfies the core semantic search requirements, but it has limitations.

First, the selected embedding model uses a fixed 384-dimensional schema. Switching to a model with a different output dimension requires changing the database schema and ORM vector dimension. This is documented in the README.

Second, the default semantic retrieval query relies on a single embedding per document. Long documents may require chunk-level embeddings to avoid losing important local meaning. The preprocessing module includes chunking support, but the AG News ingestion path stores one embedding per article because the dataset documents are short.

Third, semantic similarity alone does not guarantee factual relevance or category correctness. A hybrid search strategy that combines lexical filters, metadata constraints, and vector scores could improve precision.

Fourth, the TF-IDF baseline is built in memory. This is acceptable for a small project, but a production implementation should use PostgreSQL full-text search, OpenSearch, or another indexed lexical retrieval system.

Fifth, evaluation currently measures latency, score averages, and top-k overlap. A stronger evaluation would use human relevance judgments or labeled query-document pairs to compute precision, recall, mean reciprocal rank, and normalized discounted cumulative gain.

## Proposed Improvements

Future work should include:

- Chunk-level embedding storage for long documents.
- Hybrid ranking that combines pgvector similarity with PostgreSQL full-text search.
- Metadata filters for category, date, and source.
- Relevance-labeled evaluation queries and ranking metrics.
- A migration mechanism for multiple embedding dimensions.
- Docker Compose configuration for reproducible PostgreSQL and pgvector setup.
- Optional multilingual embeddings with `paraphrase-multilingual-MiniLM-L12-v2`.
- Comparative evaluation of multiple embedding models.

This project implements a complete semantic search pipeline using Python, Sentence-Transformers, PostgreSQL, and pgvector. ## Detailed Design Rationale

### Retrieval Requirements

The system is designed around five retrieval requirements. First, it must accept unrestricted natural-language queries rather than a fixed keyword syntax. Second, it must return ranked documents with a measurable relevance score. Third, it must persist embeddings in a vector database rather than computing all document vectors at query time. Fourth, it must provide a classical baseline so semantic retrieval can be compared against a non-neural method. Fifth, it must provide reproducible scripts for setup, ingestion, search, and evaluation.

These requirements lead to a batch-oriented ingestion design and a low-latency query design. Document embedding is performed once during ingestion because it is the expensive stage for large corpora. Query embedding is performed at runtime because each query is new. The database stores both the original document text and the embedding so the application can return human-readable results without joining against external files.

### Schema Normalization

The schema separates documents from embeddings. This is preferable to storing vectors directly on the documents table because it supports multiple embedding models for the same document. For example, the same document can have an `all-MiniLM-L6-v2` embedding and a multilingual embedding. The `model_name` column in the embeddings table identifies which vector should be used for a given query. This design also supports future model comparison without reloading the dataset.

The metadata column uses JSONB because dataset-specific attributes can vary. AG News stores the category label. Another dataset might store publication date, source URL, author, legal jurisdiction, medical specialty, or product rating. JSONB allows these fields to be preserved while the core schema remains stable.

### Indexing Strategy

The IVFFlat index is appropriate for the educational scope of this project because it is available in pgvector, simple to configure, and effective for approximate nearest-neighbor retrieval when many vectors are stored. The index uses cosine operators because both document and query embeddings are normalized and the project interprets relevance as angular similarity in embedding space.

For small datasets, a sequential scan can be fast enough and may produce exact results. For larger datasets, approximate indexes reduce search time by limiting the number of candidate vectors considered. This tradeoff can reduce recall if index parameters are not tuned. The current schema uses a conservative fixed list count suitable for a project-scale corpus. A production deployment should tune `lists` and query probes based on corpus size, latency requirements, and recall targets.

## Operational Workflow

### Environment Setup

The project uses environment variables for database configuration and model selection. The `.env.example` file documents the required values. The shell scripts activate the project virtual environment before running Python entry points. This keeps the runtime environment consistent across ingestion, search, and evaluation.

The setup workflow is intentionally sequential:

1. Install system and Python dependencies.
2. Configure PostgreSQL and pgvector.
3. Apply the schema.
4. Ingest documents and embeddings.
5. Run search or evaluation.

This order prevents common failure modes. The schema cannot be applied before the database exists. Embeddings cannot be inserted before pgvector is enabled. Search cannot return results before ingestion has populated the tables.

### Ingestion Controls

The ingestion script exposes a document limit and an embedding batch size. The limit controls experiment scale. A small limit is useful for quick functional testing. A larger limit improves retrieval coverage and evaluation realism. The batch size controls memory and throughput during embedding generation. Larger batches generally improve throughput but require more memory.

Embeddings are inserted in chunks after generation. Chunked insertion reduces transaction size and avoids keeping an unnecessarily large database transaction open. If an insertion chunk fails, the transaction context rolls back that chunk instead of leaving a partial write inside the same transaction.

### Search Interface

The search script supports both single-query and interactive modes. Single-query mode is useful for repeatable checks and demonstrations. Interactive mode is useful for qualitative exploration. The `--no-tfidf` option allows the user to skip baseline construction when only semantic retrieval is needed. This matters because TF-IDF construction loads all documents into memory and fits the vectorizer before searching.

The displayed results include rank, score, title, and snippet. The score helps compare relative ranking within one method. It should not be interpreted as an absolute relevance probability. Scores from semantic search and TF-IDF are not directly equivalent because they are computed in different vector spaces.

## Evaluation Methodology

### Query Set

The evaluation query set covers multiple AG News topics. Business queries target markets, mergers, and financial events. Science and technology queries target space missions and machine learning. World queries target elections, terrorism, and environmental issues. Sports queries target football and Olympic performance.

This query diversity helps reveal the behavior of semantic retrieval and TF-IDF across different domains. A narrow query set could make one method appear stronger than it is. For example, TF-IDF can perform very well when query terms are distinctive and present in relevant documents. Semantic retrieval can perform better when relevant documents use different wording from the query.

### Metrics

The current evaluation records latency, top-k overlap, and average score. Latency measures runtime behavior. Top-k overlap measures how similarly the two methods rank documents. Average score provides a rough comparison of score distributions within each method.

Overlap is not a relevance metric by itself. Low overlap does not mean one method is wrong; it means the methods retrieved different documents. Because semantic and lexical retrieval optimize different signals, low overlap is expected for some queries. Stronger evaluation should use manually labeled relevance judgments and ranking metrics such as precision at k, recall at k, mean reciprocal rank, and normalized discounted cumulative gain.

### Reproducibility

The evaluation writes both machine-readable and visual outputs. The CSV file supports external analysis in notebooks or spreadsheets. The PNG chart supports quick inspection of latency, overlap, and score trends. The notebook complements this by enabling deeper exploratory analysis, including category distribution and embedding visualization.

Reproducible results still depend on the selected document limit, model version, hardware, and PostgreSQL configuration. These parameters should be recorded when reporting final results. For a formal experiment, the same ingestion limit and model should be used for all methods being compared.

## Security and Reliability Considerations

### Configuration Handling

Database credentials are loaded from environment variables rather than hard-coded application constants. This is appropriate for a local project and supports different environments. The database connection code constructs the SQLAlchemy URL with structured URL creation so special characters in passwords are escaped correctly. Raw psycopg2 connections use keyword parameters for the same reason.

### Transaction Handling

Database session helpers commit on success and roll back on failure. This keeps ingestion and schema operations from silently leaving inconsistent partial writes after an exception. Search uses a session context as well, which closes the connection after the query finishes.

### Input Handling

The query text is passed as a SQL parameter, not interpolated directly into SQL. The vector value is serialized as a pgvector-compatible literal and passed through SQLAlchemy parameters. This prevents normal query text from changing SQL structure. Operational scripts should still be run in trusted environments because they perform database setup and installation tasks.

### Failure Modes

Common failure modes include missing PostgreSQL, missing pgvector, invalid credentials, no ingested documents, model download failure, and insufficient memory for a large embedding batch. The project exposes these failures through command output. The README documents the expected setup order and the need to reset the schema when changing to an embedding model with a different dimension.

## Appendix: Interpretation Guidelines

Semantic search should be interpreted as approximate meaning-based retrieval. It can retrieve documents with related context even when exact words differ, but it can also retrieve documents that are semantically broad rather than specifically relevant. TF-IDF should be interpreted as lexical relevance. It rewards exact term overlap and can be precise for distinctive terms, but it can miss paraphrases and synonyms.

The best retrieval system for production is often hybrid. A hybrid system can use lexical retrieval to enforce exact constraints and vector retrieval to capture semantic similarity. Metadata filters can then restrict results by source, category, or date. This project provides the required foundation for such a system by implementing both semantic and classical retrieval in one codebase.
